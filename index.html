Git in it - Initializes a new Git repository. If you want to place a project under
revision control, this is the first command you need to learn.
72
Git add - Moves changes from the working directory to the staging area.
This gives you the opportunity to prepare a snapshot before
committing it to the official history.
73
Git push - Pushing is the opposite of fetching (with a few caveats). It lets you
move a local branch to another repository, which serves as a
convenient way to publish contributions
74
Git commit - Takes the staged snapshot and commits it to the project history.
Combined with git add, this defines the basic workflow for all Git users.
75
Git clone - Creates a copy of an existing Git repository. Cloning is the most
common way for developers to obtain a working copy of a central
repository.
76
Git pull - Retrieves and merges the latest changes from a remote repository into your
local repository. It updates your current branch with the most recent commits
made by others, ensuring your local copy is up to date.

CONFLICTS IN THE GITHUB OR THE GITHUB DEKSTOP AND STEP TO RESOLVE THAT
83
84
GITHUB CONFLICTS - GitHub conflicts occur when multiple contributors make conflicting changes to thesame file or files
in a Git repository. These conflicts typically arise when two or more individuals modify the same
lines of code or make changes that conflict with each other.
85
86
RESOLVE THE CONFLICTS ON GIT HUB - 1. Pull the latest changes: Before merging your changes, make sure you have the latest version of
the repository by pulling the changes from the remote repository. In GitHub Desktop, select the
repository and click on "Fetch origin" to get the latest changes.
87
2. Identify the conflicts: After fetching the changes, GitHub Desktop will notify you if there are any
conflicts. Conflicted files will be listed under the "Changes" tab, marked with a red "Merge conflict"
status.
88
3. Resolve the conflicts: Click on the conflicted file in GitHub Desktop. It will open a built-in text
editor showing the conflicting sections of the code, marked by "<<<<<<< HEAD" and ">>>>>>>".
Manually edit the code to resolve the conflicts, choosing the desired changes and removing the
conflict markers.
89
4. Mark conflicts as resolved: Save the changes in the text editor. GitHub Desktop will recognize
that the conflicts have been resolved and mark the file as "Resolved" under the "Changes" tab.
90
5. Commit the changes: Provide a commit message to explain the conflict resolution. Select the
resolved files by checking the checkboxes next to them. Click on "Commit" to create a new commit.
91
6. Push the changes: Finally, click on "Push origin" in GitHub Desktop to push the resolved
changes to the remote repository.
92
93
RESOLVE THE CONFLICTS ON GIT HUB DEKSTOP - 1. **Pull the latest changes**: Open GitHub Desktop and select the repository you're working on.
Click on the "Fetch origin" button to pull the latest changes from the remote repository.
94
2. **Identify the conflicts**: After fetching the changes, GitHub Desktop will indicate if there are any
conflicts in the repository. Conflicting files will be listed under the "Changes" tab, marked with a red
"Merge conflict" status.
95
3 . **Resolve the conflicts**: To resolve conflicts, click on the conflicted file in GitHub Desktop. It will open
the built-in text editor, highlighting the conflicting sections of the code with markers such as
" <<<<<<< HEAD" and ">>>>>>>". Manually edit the code to resolve the conflicts, choosing the
desired changes and removing the conflict markers.
96
4. **Mark conflicts as resolved**: After resolving the conflicts, save the changes in the text editor.
GitHub Desktop will detect that the conflicts have been resolved and mark the file as "Resolved" under
the "Changes" tab.
97
5. **Commit the changes**: Add a commit message to explain the conflict resolution. Stage the
resolved files by checking the checkboxes next to them. Click on the "Commit" button to create a
new commit.
98
6. **Push the changes**: Finally, click on the "Push origin" button in GitHub Desktop to push the
resolved changes to the remote repository.

TYPES OF API'S -: 1. Web APIs: These APIs allow different web applications to communicate with each other over the
internet. They are used to access and interact with web-based services, enabling developers to
retrieve data from servers, send data to servers, and perform actions on remote systems.
139
2. RESTful APIs: These APIs follow a set of rules for building web services. They use standard HTTP
methods (GET, POST, PUT, DELETE) to perform operations on resources and are widely used for
scalable and interoperable web services.
140
3. SOAP APIs: SOAP APIs use an XML-based messaging protocol and allow communication between
different systems and applications. They are known for their extensive features and robustness.
141
4. GraphQL APIs: GraphQL APIs use a query language to request specific data needed by clients.
They provide flexibility, reduce over-fetching or under-fetching of data, and enhance performance.
142
5. Real-Time APIs: Real-time APIs facilitate instant data transmission and notifications, ideal for
applications requiring immediate updates, such as chat applications or real-time analytics.
143
6. Library APIs: Library APIs provide developers with pre-built functions and classes to perform specific
tasks. They help accelerate development by offering ready-made code for tasks like image processing
or machine learning.
